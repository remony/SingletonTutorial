The Singleton Pattern

The Sun, you see it everyday, and its always the same no matter where in the world you are until the night sky appears.

The Singleton pattern is a commonly design pattern where you have a single instance of a class with global access to its resources. For example in a game you would implement a singleton to provide a global access to to a games core functions or managers such as music which would play between scenes or writing logs on the console but you also want to send them to a server.

The singleton ensures that throughout the life cycle of that application/game there will only be one instance of that class until the application or game has ended.

“Ensure a class has one instance, and provide a global point of access to it”

The sun is a great example of a singleton as there is only one instance of the sun, and all the planets can access the suns variables for example light and heat.

What is a Singleton?

From Wikipedia:

In software engineering, the singleton pattern is a design pattern that restricts the instantiation of a class to one object. This is useful when exactly one object is needed to coordinate actions across the system.

Instantiating a Singleton

Singletons can only have one instance in the whole application. This is achieved via a private constructor and a getInstance() method. 

                
                    public class SampleSingleton {
                    private static SampleSingleton instance = null;
                    private SampleSingleton(  ) {

                    }
                    public static SampleSingleton getInstance(  ) {
                    if ( instance == null) instance = new SampleSingleton ( ) ;
                    return instance;
                    }
                    }

Problem with concurrency

Sometimes, when using a singleton we can find problems with concurrency. These occur when two threads simultaneously try to access the single instance returned by getInstance() .

Both threads lock the resource, and they enter into deadlock . Programming languages often provide ways around this, with syntax to lock the resource before it is used, but it can be a problem for newer programmers.

A way to prevent these concurrency problems in Java is to synchronize the instantiation of the Singleton: 

                
                    public class SampleSingleton {
                    private static SampleSingleton instance = null;
                    private SampleSingleton(  ) {

                    }
                    public static SampleSingleton getInstance(  ) {
                    if ( instance == null) {
                    synchronized ( SampleSingleton. class ) {
                    instance = new SampleSingleton ( ) ;
                    }
                    }
                    return instance;
                    }
                    }

Even this way of doing it isn’t perfectly safe, however. Another option, commonly recommended, is to eagerly initialise the singleton as a static variable: 

                
                    public class SampleSingleton {
                    private static final SampleSingleton INSTANCE = new SampleSingleton ( ) ;
                    private SampleSingleton(  ) {

                    }
                    }

And then accessing the instance like so: 

                
                    . . .
                    SampleSingleton singleton = SampleSingleton. INSTANCE;
                    . . .

The last option, to guarantee 100% of the time that the singleton will be thread-safe, is to create a static inner class: 

                
                    public static class SimpleSingleton {
                    private SimpleSingleton(  ) {
        // Private constructor prevents initialisation from other classes
    }

                    public static SimpleSingleton getInstance(  ) {
                    return SingletonHolder. INSTANCE;
                    }

                    private static class SingletonHolder {
                    private static final SimpleSingleton INSTANCE = new SimpleSingleton ( ) ;
                    }
                    }

Subclassing a singleton

Subclassing a singleton may be interesting if there are properties we want to be be able to change, while the singleton still remains a globally accessible object.

For example, if we had a FileSystem  singleton, we could subclass it to PS3FileSystem  or XboxFileSystem , depending on which platform we are using. 

                FileSystem& FileSystem:: instance(  )
                    {
                    #if PLATFORM == PLAYSTATION3
                    static FileSystem * instance = new PS3FileSystem(  ) ;
                    #elif PLATFORM == WII
                    static FileSystem * instance = new WiiFileSystem(  ) ;
                    #endif

                    return * instance;
                    }

Regretting the singleton

Many people are against singletons (and global variables as a whole). There are a few reasons for this 3 , such as:

  1. Make code harder to understand, because calling SomeClass.getGlobalData()  introduces a dependency on a function that should otherwise be self-contained.
  2. They encourage coupling: calling your global AudioEngine  singleton from a Physics  class sounds like a terrible idea, but not if you don’t know any better.
  3. The concurrency problems mentioned above.

Where should we not use Singletons?

Nowhere?

Many people will tell you to never use Singletons . They are bad, they say. They increase coupling, dependencies, make it difficult to test, they say. They have problems with multi-threading, they say.

Somewhere?

Sometimes, you will need to provide global access to a class, such as in games development, where you may have a global World  or Game  object. Sometimes you may have a global FileSystem , AudioEngine , PhysicsEngine , or other. Let’s take a look at some of these examples, and, why in some instances, although you may think using a Singleton is the best idea, it can sometimes really not be.

Problems with Singletons

A, B, and C, or friends and secrets

Lets say you have classes A  and B , which are completely unrelated. You also have a global Singleton called C .

Upon reading the code, a developer may think that A  and B  are unrelated (which they seem to be), but if they both use C , and the global state of C  affects A  and B , then they really are related by the functionality and state provided by C .

This is an increase in coupling caused by a Singleton, which can make code not perform as you expect, and also introduce increased and unexpected complexity.

Or, as Miško Hevery  puts it 4 :

Think of it another way. You can live in a society where everyone (every class) declares who their friends (collaborators) are. If I know that Joe knows Mary but neither Mary nor Joe knows Tim, then it is safe for me to assume that if I give some information to Joe he may give it to Mary, but under no circumstances will Tim get hold of it. Now, imagine that everyone (every class) declares some of their friends (collaborators), but other friends (collaborators which are singletons) are kept secret. Now you are left wondering how in the world did Tim got hold of the information you gave to Joe.

Printers and ConnectionPools

Our program will only have a pool of connections for a printer, so that we cannot overload the printer with too many requests. A Singleton will be used for this connection pool, since it will be consistent across the whole program.

Although a good idea, it seems unnecessarily limiting. If throughout all your program you can see code like this: 

                
                    public class Payroll {
                    . . .
                    public void print(  ) {
                    PrinterConnectionPool printer = PrinterConnectionPool. getInstance(  ) ;
                    printers. print(  this . toPDF(  ) ) ;
                    }
                    . . .
                    }

What will happen when you add multiple printers? What if the user has more than one printer, or if their printer supports a connection pool of 10, 100, or 500 connections?

Is it really necessary to have a Singleton to manage these connections? Can you do dependency injection, by passing it as a parameter or using something like  Guice ?

In fact, what if the print()  method above also did this, via another Singleton called BankAccount ? 

                
                    public class Payroll {
                    . . .
                    public void print(  ) {
                    BankAccount. payEmployee(  this . employee, this . amount) ;
                    PrinterConnectionPool printer = PrinterConnectionPool. getInstance(  ) ;
                    printers. print(  this . toPDF(  ) ) ;
                    }
                    . . .
                    }

This now means you cannot print a payroll without paying an Employee! All of that because it was too easy to use the Singleton BankAccount  to make the payments automatic upon printing…

You’d have to assume code authors would never do something like that, but people make mistakes. Singletons can aid in these mistakes.

Testing

As Brian Button says 5 :

Persistent state is the enemy of unit testing. One of the things that makes unit testing effective is that each test has to be independent of all the others.

If this is not true, then the order in which the tests run affects the outcome of the tests. This can lead to cases where tests fail when they shouldn’t, and even worse, it can lead to tests that pass just because of the order in which they were run.

This can hide bugs and is evil.

Web Service and global state

Mugunth Kumar 6  gives us an example of an abuse case for a Singleton. I hope nobody does this, but it’s good to know that it is a really, really bad idea:

Let me give you an example. When you develop an app that talks to a web service, the only “state” variable I could think of is your access token.

Using a Singleton to encapsulate calls to web service that remembers the access token is fine. But if you store the response strings from one method and use it in another method and then release it in yet another method, you are doing it wrong.

In this instance, using a Singleton could make your program a lot shorter, because you only have to execute a web request, and write the response to the singleton.

However, as soon as a user does more than one request, you’ll need to know which request was last executed. If you want to test a given request, you’ll need to place data in the Singleton. Now all of your unit tests will depend on executing another request first to place data in the global singleton, and thus your tests are no longer independent.

Alternatives to Singletons

Think about whether you really need the Singleton. The example below shows a project where a singleton, BulletManager  was used, but it wasn’t really necessary. 

                
                    class Bullet
                    {
                    public :
                    int getX(  ) const { return x_; }
                    int getY(  ) const { return y_; }

                    void setX(  int x) { x_ = x; }
                    void setY(  int y) { y_ = y; }

                    private :
                    int x_, y_;
                    } ;

                    class BulletManager
                    {
                    public :
                    Bullet* create(  int x, int y)
                    {
                    Bullet* bullet = new Bullet(  ) ;
                    bullet- > setX(  x) ;
                    bullet- > setY(  y) ;

                    return bullet;
                    }

                    bool isOnScreen(  Bullet& bullet)
                    {
                    return bullet. getX(  ) >= 0 &&
                    bullet. getX(  ) < SCREEN_WIDTH &&
                    bullet. getY(  ) >= 0 &&
                    bullet. getY(  ) < SCREEN_HEIGHT;
                    }

                    void move(  Bullet& bullet)
                    {
                    bullet. setX(  bullet. getX(  ) + 5 ) ;
                    }
                    } ;

In the example above, someone was using a Bullet  class to hold data about a bullet, like its x  and y  location. They were then using a BulletManager  class to move the bullet. This is an extreme example, but nevertheless worth exploring.

An obvious solution below: 

                
                    class Bullet
                    {
                    public :
                    Bullet(  int x, int y) : x_(  x) , y_(  y) { }

                    bool isOnScreen(  )
                    {
                    return x_ >= 0 && x_ < SCREEN_WIDTH &&
                    y_ >= 0 && y_ < SCREEN_HEIGHT;
                    }

                    void move(  ) { x_ + = 5 ; }

                    private :
                    int x_, y_;
                    } ;

There was no need for BulletManager  in the first place!

Use something already global

Usually, games will have an unavoidable singleton, like  Game , or  World . Often it is better to have these singletons contain things like FileSystem , AudioEngine , Log , or others.

Pass it as a parameter!

It can be cumbersome, but it can also be a good solution. Android  passes a Context  class nearly everywhere. Are our games going to be that large and complex that passing a couple parameters here and there is too difficult?

In order to pass it as a parameter everywhere, the Google Testing Blog 7  recommends using an ApplicationFactory  class, which will instantiate the necessary classes. You will only create one instance of an ApplicationFactory  class in the main()  method.

ToolBox

The Unity Alternative

As addressed on the Unity3D wiki, Singletons are generally abused by being misused a lot, so they encourage that you use ToolBox instead.

ToolBox

The ToolBox is a Singleton but improves on the concept by improving unit testing and coupling. What makes the ToolBox different from the Singleton is that a Singleton is a class where as the ToolBox is the application which contains the components required. You can learn more about the ToolBox in this paper http://www.ibm.com/developerworks/webservices/library/co-single/co-single-pdf.pdf .

The ToolBox makes unit testing easier.
